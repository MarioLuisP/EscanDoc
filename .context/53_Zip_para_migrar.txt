● La idea del ZIP es exactamente lo correcto para este caso. Simple, sin dependencias externas, sin login.

  Lo que haría:
  EXPORT:
  SQLite DB + carpeta de JPGs → ZIP → share_plus → usuario elige dónde guardar
                                                     (Drive, WhatsApp, email, USB...)

  IMPORT:
  file_picker (ya lo tenemos) → elige el ZIP → extraer → remapear paths → restaurar

  Packages:
  - archive ^4.x — ZIP creation/extraction, muy liviano, puro Dart
  - share_plus — para el share nativo (si no está ya)
  - file_picker — ya lo tenemos ✓

  El único tema no trivial: los paths en la DB son absolutos (/data/user/0/.../docs/imagen.jpg). Al restaurar en otro dispositivo cambia el path base. La solución: guardar los JPGs en el ZIP con paths relativos
  (images/imagen.jpg) y al restaurar reescribir la columna file_path con el nuevo getApplicationDocumentsDirectory().

  Tamaño estimado: depende de cuántos documentos, pero con JPGs comprimidos a ~850KB y compresión ZIP → muy manejable para Drive/email.

  Lo que NO haría:
  - Firebase/Supabase → overkill, requiere cuenta
  - iCloud nativo → iOS-only, complejo
  - Google Drive API directo → OAuth, un lío como dijiste

  En resumen: tu idea original es la mejor. Es lo que hace Obsidian, Signal, WhatsApp con sus backups.

## IMPLEMENTADO (Mayo 2026)

### Extensión y nombre
- Formato: escandoc_DD-MM-YYYY.escdc  (ej: escandoc_03-05-2026.escdc)
- Extensión propia .escdc — sigue siendo ZIP internamente
- MIME type: application/x-escdc

### Estructura del ZIP
- meta.json       → schema_version (3), export_date, count
- documents.json  → todos los campos de DocumentModel excepto id; file_path relativo (solo nombre de archivo)
- images/         → una entrada por imagen existente en disco

### Decisiones de diseño
- IMPORT es merge, no reemplazo: los documentos existentes no se tocan
- Colisiones de nombre de archivo: sufijo _bk, _bk2, _bk3... (no _imported)
- Duplicados reales (usuario re-escaneó antes de restaurar): se aceptan como comportamiento conocido, el usuario borra manualmente
- id no se incluye en documents.json → SQLite asigna auto-increment al importar → sin colisiones de PK
- export_date en meta.json en formato ISO (para parsing); el nombre del archivo usa DD-MM-YYYY (legible para el usuario)

### Capas
- BackupRepository (abstract)                                     → lib/features/backup/domain/repositories/backup_repository.dart
- BackupRepositoryImpl (archive + dart:io)                        → lib/features/backup/data/repositories/backup_repository_impl.dart
- BackupProvider (ChangeNotifier, export + importBackup)          → lib/features/backup/presentation/providers/backup_provider.dart
- Tests (14 en total, repository + provider)                      → test/features/backup/

### UI
- Acceso: Configuración → debajo de "Avisos de vencimiento" → dos cards: "Guardar backup" / "Restaurar backup"
- Export: share_plus con ShareParams(mimeType: application/x-escdc) → share sheet del OS
- Import: file_picker acepta .escdc y .zip (fallback para backups viejos) → _runBackupImport()
- Sharing intent: home_page._processSharedFile detecta extensión escdc → _runBackupImport() (mismo path que file_picker)
- _runBackupImport existe en home_page (para intents) y en settings_page (para file_picker)

### Registro de tipo de archivo
- Android: intent-filters en AndroidManifest para ACTION_SEND y ACTION_VIEW con mimeType application/x-escdc
- iOS: UTExportedTypeDeclarations + CFBundleDocumentTypes en Info.plist (UTI: com.passalia.escandoc.backup, conforma a public.zip-archive)

### Fuera de scope (v1)
- "Abrir con" desde iOS (receive_sharing_intent en iOS postergado, igual que el resto del plugin)
- Deduplicación automática de duplicados reales
- Backup automático / backup incremental
