● La idea del ZIP es exactamente lo correcto para este caso. Simple, sin dependencias externas, sin login.

  Lo que haría:
  EXPORT:
  SQLite DB + carpeta de JPGs → ZIP → share_plus → usuario elige dónde guardar
                                                     (Drive, WhatsApp, email, USB...)

  IMPORT:
  file_picker (ya lo tenemos) → elige el ZIP → extraer → remapear paths → restaurar

  Packages:
  - archive ^4.x — ZIP creation/extraction, muy liviano, puro Dart
  - share_plus — para el share nativo (si no está ya)
  - file_picker — ya lo tenemos ✓

  El único tema no trivial: los paths en la DB son absolutos (/data/user/0/.../docs/imagen.jpg). Al restaurar en otro dispositivo cambia el path base. La solución: guardar los JPGs en el ZIP con paths relativos
  (images/imagen.jpg) y al restaurar reescribir la columna file_path con el nuevo getApplicationDocumentsDirectory().

  Tamaño estimado: depende de cuántos documentos, pero con JPGs comprimidos a ~850KB y compresión ZIP → muy manejable para Drive/email.

  Lo que NO haría:
  - Firebase/Supabase → overkill, requiere cuenta
  - iCloud nativo → iOS-only, complejo
  - Google Drive API directo → OAuth, un lío como dijiste

  En resumen: tu idea original es la mejor. Es lo que hace Obsidian, Signal, WhatsApp con sus backups.

● Bastante lejos, sin login. Acá el mapa completo:

  Sin login (todo posible):
  Crear ZIP
      ↓
  Dialog: ¿Qué hacés con él?
      ├── "Guardar en carpeta"  → file_picker.saveFile()
      │                            Android: abre SAF* → Drive aparece como destino nativo
      │                            iOS: abre Files → iCloud Drive disponible
      │
      └── "Compartir"          → share_plus share sheet
                                   Android/iOS: todas las apps instaladas
                                   (Drive, Dropbox, Telegram, Gmail, WhatsApp...)

  * SAF (Storage Access Framework) es el file picker del sistema Android — Drive está integrado como ubicación nativa sin ningún OAuth. El usuario simplemente navega a "Mi unidad" y guarda. En iOS pasa lo mismo con
  Files/iCloud.

  Con login (fuera de scope):
  - Subir a una carpeta específica de Drive programáticamente
  - Backup automático sin intervención del usuario
  - Listar/comparar versiones en Drive

  ---
  La frontera práctica sin login:

  El usuario toca "Backup", la app crea el ZIP, muestra el dialog, y el usuario elige guardar directo en Drive (vía SAF/Files) o compartirlo por cualquier app. No hay diferencia visible respecto a tener integración nativa — el
  OS hace el trabajo.

  Lo único que no podés hacer sin login es disparar la subida desde la app sin que el usuario toque algo. Para todo lo demás, el OS te cubre.
 y para recuperarlo?  cual seria el flujo mas facil para el usuario?
● El flujo más fácil para el usuario es que no tenga que entrar a la app primero:

  Usuario abre Drive/Files/WhatsApp                                                                                                                                                                                                      → busca "escandoc_backup_2026-03-01.zip"
      → toca "Abrir con" / "Compartir"
      → elige EscanDoc
      → la app recibe el ZIP (receive_sharing_intent, igual que las imágenes)
      → detecta que es un backup (por nombre o contenido)
      → dialog: "¿Restaurar backup del 1/3/2026?
                 Esto reemplazará todos tus documentos actuales."
      → confirma → restaura → recarga

  ¿Por qué este flujo?
  - El usuario ya está donde están sus archivos
  - "Abrir con" es más intuitivo que entrar a la app y navegar a Settings
  - Ya tenemos receive_sharing_intent configurado — solo habría que agregar ZIP a los tipos aceptados

  Como fallback (usuarios que no saben el "Abrir con"):
  - Settings → Restaurar backup → file_picker → elige el ZIP

  Un detalle importante para el ZIP:
  Ponerle fecha en el nombre: escandoc_backup_2026-03-01.zip → el usuario sabe exactamente qué está restaurando sin abrir nada.

  Lo único delicado: avisar bien que se reemplazan los documentos actuales. Para personas mayores, ese dialog tiene que ser muy claro — quizás con dos pasos de confirmación.
